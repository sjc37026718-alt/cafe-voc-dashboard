## PART 2. 해커톤 문제 풀이 프레임워크

### 핵심 원칙: "도구보다 사고가 먼저, 빼는 게 실력"

#### 흔한 실패 원인
| 증상 | 진짜 원인 |
|------|----------|
| 시간 안에 못 끝냄 | 다 만들려다 스코프 폭주 |
| 엉뚱한 걸 만듦 | 진짜 문제를 정의하지 않음 |
| 복잡해서 꼬임 | 필요 없는 걸 미리 만듦 |

### 문제를 받으면 — 4단계 (5분이면 충분)

| 단계 | 스스로 묻기 | 예시 |
|------|-----------|------|
| 1. 진짜 문제 | 표면 요청 말고, 본질이 뭔가? 한 문장으로 | "의견이 너무 많아 다 못 본다" |
| 2. 누가/무엇 | 누가 쓰고 무엇을 얻나? | "PM이 주요 불만 top3를 1분에 파악" |
| 3. 가장 단순한 해법 | 이거 없이도 되나? 뺄 건? | "자동 수집/실시간은 빼고, 붙여넣으면 분류" |
| 4. 핵심 한 조각 | 8시간에 보여줄 단 하나는? | "붙여넣기 → 분류/top3 화면" |

**클로드에게 바로 시킬 수 있는 프롬프트:**
```
이 문제를 위 4단계로 정리하고, 8시간에 가능한 가장 단순한 버전을 제안해줘.
```

### 엔지니어링적 사고 5원칙

1. **입력 → 처리 → 출력으로 쪼갠다** — 막막할 때 이 셋만 적기
2. **가장 단순한 것부터** — 지금 필요 없으면 안 만든다
3. **동작하는 것 → 좋은 것** — 못생겨도 먼저 돌아가게
4. **하드코딩도 괜찮다** — 데모는 진짜 자동화가 아니어도 됨
5. **위험한 곳부터 검증** — 불확실한 부분 먼저 1분 테스트

### 오버엔지니어링 경계

| 욕심 (과함) | 현실 (충분) |
|------------|-----------|
| 실시간 자동 처리 | 버튼 누르면 실행 |
| 모든 경우 다 처리 | 대표 1~2가지만 |
| DB/로그인/배포 다 | 동작하는 핵심 화면 하나 |
| 멀티에이전트/복잡한 구조 | 그냥 Claude 한 명에게 |

## PART 3. PRD → 시각화 → MVP 흐름

### 흐름 요약
```
문제 파악 → PRD 작성 → 시각화 → MVP 제작 → 발표
   (왜)      (무엇을)    (어떻게 보일지)  (실제로 돌아가는 것)
```

### PRD 작성 프롬프트
```
기업: [기업명]
문제: [해결할 문제]
목표: [만들 솔루션]

위 내용을 기반으로 PRD를 작성해줘. 다음 항목을 포함해:
- 문제 정의 (Problem Statement)
- 목표 사용자 (Target User)
- 핵심 기능 (Key Features)
- 성공 지표 (Success Metrics)
- 우선순위 (Must-have / Nice-to-have)
```

### 시각화 프롬프트
```
위 PRD를 바탕으로 [대시보드/랜딩페이지/플로우차트]를
HTML로 만들어줘. 차트는 Chart.js, 스타일은 Tailwind CSS.
```

### MVP 프롬프트
```
위 PRD와 시각화를 바탕으로 MVP를 HTML 단일 파일로 만들어줘.
기술 스택: HTML + Tailwind CSS + Chart.js
데이터: 현실적인 더미 데이터로 채워줘
```

## PART 4. 도구 선택 가이드

### 기본값은 "그냥 한 대화"
실무 문제의 80~90%는 클로드에게 한 대화로 시키면 끝남.

| 상황 | 이렇게 | 왜 |
|------|--------|-----|
| 한 번만 하는 작업 | 그냥 한 대화 | 가장 빠르고 쌈 |
| 같은 작업을 매번 반복 | Skill로 저장 | /이름으로 재사용 |
| 독립적인 여러 단계 | 서브에이전트로 분담 | 갈래가 다를 때만 |
| 내 업무 도구 연결 | Cowork(커넥터) | 노션/드라이브/슬랙 |

### 배포 도구 비교

| | Vercel | Streamlit |
|--|--------|-----------|
| 뭘 배포하나 | 웹사이트, 랜딩페이지 (HTML/JS) | Python 데이터 앱, 대시보드 |
| 적합한 MVP | 캠페인 페이지, 인터랙티브 UI | 데이터 분석 대시보드, AI 챗봇 |
| 난이도 | 중 | 하 (Python만 알면 됨) |

- GitHub: 있으면 편하지만 필수 아님
- Supabase: 해커톤 MVP 수준에서는 대부분 필요 없음

### 핵심: 생성/분류/분석은 앱이 아니라 Claude가 직접 한다
"앱이 AI를 부른다"(API 키 필요) ❌
"Claude가 직접 만든다"(키 불필요) ⭐

## PART 5. 프롬프트 템플릿 모음

### 기업 유형별 프롬프트

**B2C (큐피스트, 엔트레씨, 키즐링 등)**
```
너는 [앱명]의 그로스 마케터야.
[타겟]의 [지표]가 [현재 수치]로 낮아.
[목표 수치]로 올리기 위한 [전략 유형]을 설계해줘.
[구체적 채널/방법]의 타이밍과 메시지를 포함해.
```

**B2B SaaS (리뉴어스랩, Amuse8, 딥라이트 등)**
```
[제품명]의 ICP(Ideal Customer Profile)를 정의하고,
[채널 3개]로 월 [목표]개 리드를 확보하는
인바운드/아웃바운드 믹스 전략을 짜줘.
```

**브랜딩 (금화, 서스테이너블랩 등)**
```
[브랜드]는 [현재 이미지] 이미지가 강해.
[새 타겟]에게 어필하면서도 기존 고객을 잃지 않는
리브랜딩 캠페인을 제안해줘. 레퍼런스 사례도 분석해.
```

### 좋은 프롬프트 공식
```
[역할 지정] + [기업/제품 맥락] + [목표/제약조건] + [원하는 출력 형식]
```

### 프롬프트 기법 4가지

1. **역할/맥락 먼저 (RCO)**: "너는 ~야. ~를 위해. ~형식으로."
2. **분류는 예시 2~3개 (Few-shot)**: 정답 예시를 먼저 보여주기
3. **출력 형식 콕 집기**: JSON, 표, 글자수, 채널별 등
4. **여러 줄은 배치로**: CSV 통째로 주고 한번에 처리

## PART 9. 데모 시연법

- 시작은 문제 한 문장: "OO가 OO를 못 해서 만들었어요."
- 실제 입력 → 실제 출력을 한 번 시연 (더미 데이터 OK)
- 못한 부분은 솔직하게 + "이렇게 확장한다" 한 줄

### 자가 점검
| 질문 | OK 기준 |
|------|--------|
| 동작하나? | 핵심 한 기능이 더미로 끝까지 돌아간다 |
| 문제와 맞나? | 처음 정의한 '진짜 문제'를 푸는 것 |
| 보여줄 수 있나? | 입력→출력을 1분 안에 시연 가능 |
| 과하지 않나? | 안 쓰는 기능/도구가 없다 |

### 발표에서 "확장 가능성" 말하는 법
못 만든 부분은 실패가 아니라 **"확장 포인트"**로 발표하면 강점이 된다.

**공식:**
> "현재는 [지금 구현한 것]이고, 다음 단계로는 [확장 방향]이 가능합니다."

**예시 문구 모음:**
| 지금 구현한 것 | 확장 발표 문구 |
|--------------|--------------|
| 수동 입력 폼 | "앱스토어 리뷰 API로 자동 수집하면 입력 없이도 실시간 대시보드가 됩니다" |
| 키워드 기반 분류 | "AI 감정 분석 모델을 붙이면 분류 정확도를 90% 이상으로 올릴 수 있습니다" |
| JSON 파일 저장 | "Supabase DB로 전환하면 다중 매장, 다중 사용자 동시 사용이 가능합니다" |
| 로컬 실행 | "Streamlit Cloud/Vercel로 배포하면 사장님이 폰으로도 확인 가능합니다" |
| 더미 데이터 | "실제 데이터를 연결하면 바로 실무 투입 가능한 수준입니다" |
| 단일 매장 | "프랜차이즈 구조로 확장하면 본사에서 전 매장 VoC를 한눈에 볼 수 있습니다" |

**핵심:** 다 만들려다 망하는 것보다, 핵심만 만들고 나머지는 "이렇게 확장한다"고 말하는 게 훨씬 설득력 있다.

## PART 10. 해커톤 문제 접근 마인드셋

### 기업이 내는 문제는 두 가지 유형이다
| 유형 | 기업 의도 | 내가 보여줘야 할 것 |
|------|----------|-------------------|
| 이미 해결한 문제 | 기업의 자부심 → "다른 방식으로 풀 수 있나?" | 기존과 다른 관점, 더 효율적/창의적 접근 |
| 아직 해결 못한 문제 | 비용/시간 한계 → "어떻게 접근하는지 보고 싶다" | 문제 정의력 + 실행 과정 + 현실적 해법 |

→ 어떤 유형이든 **"어떻게 접근하고 풀어나갔는지"가 핵심 평가 포인트**

### 도구 사용 분업
| 작업 유형 | 사용 도구 |
|----------|----------|
| 판단/글쓰기/분석 (비코딩) | **Cowork** |
| 코드가 필요한 구현 | **Claude Code (바이브 코딩)** |

### 문제 풀이 구조화
- 문제를 한번에 풀지 말고 **끊어가며** 진행
- 각 단계에서 "어떤 형식으로 풀고 → 어떻게 이어서 만들지" 구조를 먼저 설계
- 구조가 잡히면 각 조각을 순서대로 실행

## PART 11. 클로드 사용 시 주의사항

- **최신 데이터를 모름** — 특정 기업 최근 실적, 뉴스는 직접 검색해서 붙여넣기
- **숫자를 만들어낼 수 있음** — 시장 규모, 전환율 등 구체적 수치는 반드시 검증
- **한번에 완벽을 기대하지 말 것** — 초안 → 피드백 → 수정의 반복이 핵심
- **여러 관점 요청** — "이 전략의 약점은?", "CMO 입장에서 반론해봐"
- **토큰 관리** — 작업 바뀌면 /clear, 길어지면 /compact, /context로 확인
